## 3. インフラストラクチャー

### 3.1 クラウド基盤

- **アプリケーション**: Vercel（Next.js 16 App Router）
  - リージョン: 東京 (Serverless Function) / グローバルエッジ
- **データベース / 認証 / ストレージ**: Supabase (PostgreSQL 15)
  - プライマリリージョン: AWS ap-northeast-1（東京）
  - 物理的な顧客データ保管は東京リージョンに限定

### 3.2 ネットワーク

- HTTPS (TLS 1.2+) による全通信経路の暗号化
- HTTP Strict Transport Security (HSTS): max-age=63072000; preload
- Content Security Policy: default-src 'self' ベースの厳格ポリシー
- Permissions-Policy による不要権限の明示的無効化
- Cloudflare / Vercel Edge による DDoS 緩和
- 将来的な WAF 導入計画あり

### 3.3 データセンター

Supabase のインフラ基盤は AWS 上で稼働しており、AWS のデータセンターは
以下の認証・準拠を取得しています:

- ISO 27001, ISO 27017, ISO 27018
- SOC 1 / SOC 2 / SOC 3
- PCI DSS
- FedRAMP (Moderate)

## 4. データ保護

### 4.1 暗号化

| 保護対象 | 暗号化方式 |
|---|---|
| 通信経路 | TLS 1.2+ (ALPN, HTTP/2 対応) |
| データベース保存時 | AES-256 (Supabase at-rest encryption) |
| オブジェクトストレージ | AES-256 (Supabase Storage at-rest encryption) |
| バックアップ | AES-256 |
| 認証パスワード | bcrypt による一方向ハッシュ |
| Secret | 環境変数 / 暗号化された Secret Manager |

### 4.2 キー管理

暗号化鍵はクラウドプロバイダの KMS により管理され、定期ローテーション
が行われます。

### 4.3 データ所在地

顧客データは、契約で別段の定めがない限り、東京リージョン内で保管され
ます。アプリケーション処理は東京リージョンで行われますが、Vercel のエッ
ジ配信によりリクエストはグローバルエッジポイントでも処理され得ます。た
だし、永続データは東京に固定されます。

### 4.4 マルチテナント分離

- PostgreSQL Row Level Security (RLS) を全テーブルに適用
- 全テーブルに `tenant_id` カラムを持たせ、JWT クレームの `tenant_id`
  とのマッチングを必須化
- `super_admin` ロールによるクロステナントアクセスは、監査ログで常時
  追跡

## 5. アクセス制御

### 5.1 顧客向け

- 4 段階のロール（super_admin / admin / manager / member）に基づく
  RBAC
- 最小権限の原則
- 管理者操作は監査ログ対象
- 日報閲覧ポリシーを 3 段階（manager_only / team / tenant_all）で切替
  可能
- テンプレート単位での閲覧範囲の上書き

### 5.2 認証

- メール + パスワード（パスワード要件: 長さ 12 文字以上、複雑性要件）
- Google OAuth
- パスワードリセットフロー（期限付きトークン）
- セッション管理: Supabase Auth 発行 JWT
- 将来的に TOTP / WebAuthn による MFA 強制機能を導入予定

### 5.3 社内向け（当社役職員）

- すべての本番環境アクセスに MFA 強制
- 最小権限のロール設計
- 休眠アカウントの自動無効化
- SSO 統合（Google Workspace）
- 退職時の速やかな権限剥奪

## 6. 監査ログ

### 6.1 記録対象

- 認証イベント（成功 / 失敗）
- 特権操作（ユーザー管理、テナント管理、ロール変更、招待発行、削除等）
- 決済・請求関連操作（請求書の発行、消込、無効化）
- 統合設定の変更（外部連携、Webhook 等）
- セキュリティ関連の設定変更

### 6.2 保存と保全

- 最低 1 年間の保存（法定要請があるものは法定期間）
- 改ざん防止を考慮した設計（書き込み専用チャネル）
- 顧客の求めに応じた開示（Enterprise プラン）

## 7. 脆弱性管理

### 7.1 プロアクティブ対策

- SCA (Software Composition Analysis): 依存ライブラリの継続的な脆弱性
  スキャン
- SAST (静的解析) による安全でないパターンの自動検知（計画中）
- CI パイプラインでの脆弱性ゲート
- 第三者機関による年次ペネトレーションテスト（計画中）
- Bug Bounty プログラム（導入検討中）

### 7.2 既知の対策済み脅威

コードベースには以下の脅威への明示的な対策および単体テストが存在しま
す。

- SQL インジェクション（Drizzle パラメータバインド、RLS）
- XSS（CSP、React 標準のエスケープ）
- CSRF（Same-origin ポリシー、SameSite Cookie）
- SSRF（Webhook URL 検証、プライベート IP 拒否）
- CSV Formula Injection（エクスポート時の `=` 等先頭文字エスケープ）
- Open Redirect（リダイレクト先ホワイトリスト）
- 全文検索クエリのメタ文字エスケープ
- 並行処理の二重発火（PostgreSQL advisory lock）

### 7.3 パッチ適用

| 深刻度 | 適用期限 |
|---|---|
| Critical | 48 時間以内 |
| High | 7 日以内 |
| Medium | 30 日以内 |
| Low | 90 日以内 |
