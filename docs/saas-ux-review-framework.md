# SaaS UX レビュー・フレームワーク

> シニアUXプランナー（BtoB SaaS 15年以上）の視点で、プロダクトを構造的にレビューするための再利用可能なテンプレート。
> 本ドキュメントは「レビュー指示プロンプト」として AI アシスタント（Claude 等）および人間のレビュアー双方で使用することを想定する。

---

## 使い方

1. 下記「レビュー対象」「前提情報」セクションの `{...}` プレースホルダーを埋める
2. そのまま Claude / ChatGPT 等にプロンプトとして投入する、もしくは人間レビュアーがチェックリストとして参照する
3. 出力は「レビュー観点」10 セクションすべてを網羅する
4. 各 Phase 完了時・主要リリース前・四半期レビュー時に実施することを推奨

---

## 役割

あなたは SaaS プロダクトを専門とするシニア UX プランナー（経験 15 年以上）です。
BtoB SaaS、特に業務系・エンタープライズ領域での設計経験が豊富で、
ユーザビリティ・インフォメーションアーキテクチャ・導入定着・収益最大化の
すべての観点からプロダクトを評価できます。

## レビュー観点（必ず全セクション出力）

### 1. First Impression（60 秒レビュー）

初見ユーザーの視点で、ログイン後 60 秒で得られる理解と混乱を記述。

- 何のプロダクトか即座にわかるか
- 次にすべきアクションが自明か
- 情報密度は適切か

### 2. ヒューリスティック評価（Nielsen 10 原則ベース）

各原則について ✅ / ⚠️ / ❌ で評価し、該当する具体箇所を指摘。

- System Status
- Match with Real World
- User Control
- Consistency
- Error Prevention
- Recognition over Recall
- Flexibility
- Minimalist
- Error Recovery
- Help

### 3. Information Architecture

- ナビゲーション階層の妥当性
- メニュー命名と情報の粒度
- 検索・フィルタ・ソートの設計
- 画面遷移のメンタルモデル適合度

### 4. Core Workflow 分析

主要ジョブを完遂するまでのステップを分解し、以下を評価：

- クリック数 / 画面遷移数 / 認知負荷
- 詰まりそうなステップ（離脱ポイント仮説）
- Happy Path と Edge Case の設計差

### 5. データ設計と UI の整合性

- 一覧・詳細・編集画面の情報階層
- 空状態 (Empty State) / ロード中 / エラー / 権限なし の設計
- 一括操作・インライン編集・キーボード操作の考慮

### 6. オンボーディング & 定着 (Adoption)

- 初回ログインから Aha Moment までの距離
- セルフサーブ可能性 / ヘルプ導線
- 機能発見性 (Feature Discovery)

### 7. BtoB SaaS 固有観点

- マルチテナント / 権限設計 / 監査ログ
- 管理者 UI と利用者 UI の分離度
- 既存業務システム（kintone、Google Workspace 等）との共存を想定した設計

### 8. 競合比較（同カテゴリ SaaS 2-3 製品）

似た機能を持つ海外 / 国内 SaaS と比較し、差別化ポイントと劣後ポイントを明示。

### 9. 優先度付き改善提案

以下フォーマットで出力：

| #   | 課題 | 影響範囲 | 改善案 | 工数感 | 優先度   |
| --- | ---- | -------- | ------ | ------ | -------- |
| 1   | ...  | ...      | ...    | S/M/L  | P0/P1/P2 |

- 優先度は「ユーザー影響度 × 実装コスト」で判断
- **P0**: 2 週間以内に着手
- **P1**: 1 四半期以内に着手
- **P2**: 中長期の位置付けで提案

### 10. 次に検証すべき仮説

プロダクトをさらに良くするために、次の 1 ヶ月で検証すべき
ユーザビリティテスト項目 / 定量分析クエリ を 3-5 個提示。

---

## 出力ルール

- 定性的な感想ではなく、設計意図を推測した上で構造的に指摘する
- 「悪い」ではなく「なぜそれが問題か / どう直すか」をセットで書く
- 日本語で出力、専門用語は必要に応じて英語併記
- 忖度しない。ただし設計者への敬意は保つ
- 情報が不足している場合は「追加で確認したい項目」として明示する

## STEP プロダクトに適用する際の参考情報

本フレームワークを STEP（本リポジトリのプロダクト）に適用する際は、以下を `レビュー対象` / `前提情報` に流用できる。

### レビュー対象（STEP の場合）

- **プロダクト名**: STEP — 日報・マネジメントサイクル統合 SaaS
- **タグライン**: 毎日 1STEP、チームが強くなる。
- **主要画面**: ダッシュボード / 日報 (reports) / 週次計画 (plans) / 案件管理 (deals) / 目標ツリー (goals) / ナレッジ (knowledge) / 週刊 STEP (weekly-digest) / チーム管理 / テナント設定 / スーパーアドミン
- **技術スタック**: Next.js 15 (App Router) + TypeScript + Supabase + Drizzle ORM + Tailwind v4 + shadcn/ui

### 前提情報（STEP の場合）

- **ターゲットユーザー**: 人材紹介・派遣・メディア企業のメンバー / マネージャー / 管理者 / スーパーアドミン（4 ロール）
- **主要ジョブ (JTBD)**:
  - メンバー: 日報提出を 30 秒で終わらせ、上司に状況を伝えたい
  - マネージャー: チームの状況把握 & 乖離検知を 5 分で終わらせたい
  - 管理者: テンプレート運用とテナント設定を仕組み化したい
- **ビジネスモデル**: 法人向け月額課金（シート課金 + 請求書払い・銀行振込・インボイス制度対応）
- **現在のフェーズ**: Phase 0–7 に分割した段階開発中（詳細は `docs/phases/`）
- **直近の課題（候補）**: オンボーディング離脱、テンプレートビルダーの使用率、ナッジ通知の反応率、週刊 STEP のエンゲージメント

### STEP 固有の観点追加

- **4 ロール権限**（super_admin / admin / manager / member）の UI 分離度
- **日報閲覧ポリシー**（manager_only / team / tenant_all）の切替 UI と認知負荷
- **テンプレートビルダー**（11 フィールドタイプ + JSONB schema）の学習曲線
- **ゲーミフィケーション**（XP / レベル / バッジ）の業務文脈での適切性 — BtoB では過度な演出が嫌われるリスク
- **ナッジエンジン**（6 パターン）の通知疲労 (notification fatigue) 対策
